     5‚Üí# Hooks reference
     6‚Üí
     7‚Üí> Reference for Claude Code hook events, configuration schema, JSON input/output formats, exit codes, async hooks, HTTP hooks, prompt hooks, and MCP tool hooks.
     8‚Üí
     9‚Üí<Tip>
    10‚Üí  For a quickstart guide with examples, see [Automate workflows with hooks](/en/hooks-guide).
    11‚Üí</Tip>
    12‚Üí
    13‚ÜíHooks are user-defined shell commands, HTTP endpoints, or LLM prompts that execute automatically at specific points in Claude Code's lifecycle. Use this reference to look up event schemas, configuration options, JSON input/output formats, and advanced features like async hooks, HTTP hooks, and MCP tool hooks. If you're setting up hooks for the first time, start with the [guide](/en/hooks-guide) instead.
    14‚Üí
    15‚Üí## Hook lifecycle
    16‚Üí
    17‚ÜíHooks fire at specific points during a Claude Code session. When an event fires and a matcher matches, Claude Code passes JSON context about the event to your hook handler. For command hooks, input arrives on stdin. For HTTP hooks, it arrives as the POST request body. Your handler can then inspect the input, take action, and optionally return a decision. Some events fire once per session, while others fire repeatedly inside the agentic loop:

    25‚ÜíThe table below summarizes when each event fires. The [Hook events](#hook-events) section documents the full input schema and decision control options for each one.
    26‚Üí
    27‚Üí| Event                | When it fires                                                                                               |
    28‚Üí| :------------------- | :---------------------------------------------------------------------------------------------------------- |
    29‚Üí| `SessionStart`       | When a session begins or resumes                                                                            |
    30‚Üí| `UserPromptSubmit`   | When you submit a prompt, before Claude processes it                                                        |
    31‚Üí| `PreToolUse`         | Before a tool call executes. Can block it                                                                   |
    32‚Üí| `PermissionRequest`  | When a permission dialog appears                                                                            |
    33‚Üí| `PostToolUse`        | After a tool call succeeds                                                                                  |
    34‚Üí| `PostToolUseFailure` | After a tool call fails                                                                                     |
    35‚Üí| `Notification`       | When Claude Code sends a notification                                                                       |
    36‚Üí| `SubagentStart`      | When a subagent is spawned                                                                                  |
    37‚Üí| `SubagentStop`       | When a subagent finishes                                                                                    |
    38‚Üí| `Stop`               | When Claude finishes responding                                                                             |
    39‚Üí| `TeammateIdle`       | When an [agent team](/en/agent-teams) teammate is about to go idle                                          |
    40‚Üí| `TaskCompleted`      | When a task is being marked as completed                                                                    |
    41‚Üí| `ConfigChange`       | When a configuration file changes during a session                                                          |
    42‚Üí| `WorktreeCreate`     | When a worktree is being created via `--worktree` or `isolation: "worktree"`. Replaces default git behavior |
    43‚Üí| `WorktreeRemove`     | When a worktree is being removed, either at session exit or when a subagent finishes                        |
    44‚Üí| `PreCompact`         | Before context compaction                                                                                   |
    45‚Üí| `SessionEnd`         | When a session terminates                                                                                   |

    47‚Üí### How a hook resolves
    48‚Üí
    49‚ÜíTo see how these pieces fit together, consider this `PreToolUse` hook that blocks destructive shell commands. The hook runs `block-rm.sh` before every Bash tool call:
    50‚Üí
    51‚Üí```json  theme={null}
    52‚Üí{
    53‚Üí  "hooks": {
    54‚Üí    "PreToolUse": [
    55‚Üí      {
    56‚Üí        "matcher": "Bash",
    57‚Üí        "hooks": [
    58‚Üí          {
    59‚Üí            "type": "command",
    60‚Üí            "command": ".claude/hooks/block-rm.sh"
    61‚Üí          }
    62‚Üí        ]
    63‚Üí      }
    64‚Üí    ]
    65‚Üí  }
    66‚Üí}
    67‚Üí```
    68‚Üí
    69‚ÜíThe script reads the JSON input from stdin, extracts the command, and returns a `permissionDecision` of `"deny"` if it contains `rm -rf`:
    70‚Üí
    71‚Üí```bash  theme={null}
    72‚Üí#!/bin/bash
    73‚Üí# .claude/hooks/block-rm.sh
    74‚ÜíCOMMAND=$(jq -r '.tool_input.command')
    75‚Üí
    76‚Üíif echo "$COMMAND" | grep -q 'rm -rf'; then
    77‚Üí  jq -n '{
    78‚Üí    hookSpecificOutput: {
    79‚Üí      hookEventName: "PreToolUse",
    80‚Üí      permissionDecision: "deny",
    81‚Üí      permissionDecisionReason: "Destructive command blocked by hook"
    82‚Üí    }
    83‚Üí  }'
    84‚Üíelse
    85‚Üí  exit 0  # allow the command
    86‚Üífi
    87‚Üí```
    88‚Üí

    95‚Üí<Steps>
    96‚Üí  <Step title="Event fires">
    97‚Üí    The `PreToolUse` event fires. Claude Code sends the tool input as JSON on stdin to the hook:
    98‚Üí
    99‚Üí    ```json  theme={null}
   100‚Üí    { "tool_name": "Bash", "tool_input": { "command": "rm -rf /tmp/build" }, ... }
   101‚Üí    ```
   102‚Üí  </Step>
   103‚Üí
   104‚Üí  <Step title="Matcher checks">
   105‚Üí    The matcher `"Bash"` matches the tool name, so `block-rm.sh` runs. If you omit the matcher or use `"*"`, the hook runs on every occurrence of the event. Hooks only skip when a matcher is defined and doesn't match.
   106‚Üí  </Step>
   107‚Üí
   108‚Üí  <Step title="Hook handler runs">
   109‚Üí    The script extracts `"rm -rf /tmp/build"` from the input and finds `rm -rf`, so it prints a decision to stdout:
   110‚Üí
   111‚Üí    ```json  theme={null}
   112‚Üí    {
   113‚Üí      "hookSpecificOutput": {
   114‚Üí        "hookEventName": "PreToolUse",
   115‚Üí        "permissionDecision": "deny",
   116‚Üí        "permissionDecisionReason": "Destructive command blocked by hook"
   117‚Üí      }
   118‚Üí    }
   119‚Üí    ```
   120‚Üí
   121‚Üí    If the command had been safe (like `npm test`), the script would hit `exit 0` instead, which tells Claude Code to allow the tool call with no further action.
   122‚Üí  </Step>
   123‚Üí
   124‚Üí  <Step title="Claude Code acts on the result">
   125‚Üí    Claude Code reads the JSON decision, blocks the tool call, and shows Claude the reason.
   126‚Üí  </Step>
   127‚Üí</Steps>
   128‚Üí

   130‚Üí
   131‚Üí## Configuration
   132‚Üí
   133‚ÜíHooks are defined in JSON settings files. The configuration has three levels of nesting:
   134‚Üí
   135‚Üí1. Choose a [hook event](#hook-events) to respond to, like `PreToolUse` or `Stop`
   136‚Üí2. Add a [matcher group](#matcher-patterns) to filter when it fires, like "only for the Bash tool"
   137‚Üí3. Define one or more [hook handlers](#hook-handler-fields) to run when matched
   138‚Üí
   139‚ÜíSee [How a hook resolves](#how-a-hook-resolves) above for a complete walkthrough with an annotated example.
   140‚Üí
   141‚Üí<Note>
   142‚Üí  This page uses specific terms for each level: **hook event** for the lifecycle point, **matcher group** for the filter, and **hook handler** for the shell command, HTTP endpoint, prompt, or agent that runs. "Hook" on its own refers to the general feature.
   143‚Üí</Note>
   144‚Üí
   145‚Üí### Hook locations
   146‚Üí
   147‚ÜíWhere you define a hook determines its scope:
   148‚Üí
   149‚Üí| Location                                                   | Scope                         | Shareable                          |
   150‚Üí| :--------------------------------------------------------- | :---------------------------- | :--------------------------------- |
   151‚Üí| `~/.claude/settings.json`                                  | All your projects             | No, local to your machine          |
   152‚Üí| `.claude/settings.json`                                    | Single project                | Yes, can be committed to the repo  |
   153‚Üí| `.claude/settings.local.json`                              | Single project                | No, gitignored                     |
   154‚Üí| Managed policy settings                                    | Organization-wide             | Yes, admin-controlled              |
   155‚Üí| [Plugin](/en/plugins) `hooks/hooks.json`                   | When plugin is enabled        | Yes, bundled with the plugin       |
   156‚Üí| [Skill](/en/skills) or [agent](/en/sub-agents) frontmatter | While the component is active | Yes, defined in the component file |
   157‚Üí
   158‚ÜíFor details on settings file resolution, see [settings](/en/settings). Enterprise administrators can use `allowManagedHooksOnly` to block user, project, and plugin hooks. See [Hook configuration](/en/settings#hook-configuration).
   159‚Üí
   160‚Üí### Matcher patterns
   161‚Üí
   162‚ÜíThe `matcher` field is a regex string that filters when hooks fire. Use `"*"`, `""`, or omit `matcher` entirely to match all occurrences. Each event type matches on a different field:
   163‚Üí
   164‚Üí| Event                                                                                           | What the matcher filters  | Example matcher values                                                             |
   165‚Üí| :---------------------------------------------------------------------------------------------- | :------------------------ | :--------------------------------------------------------------------------------- |
   166‚Üí| `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`                          | tool name                 | `Bash`, `Edit\|Write`, `mcp__.*`                                                   |
   167‚Üí| `SessionStart`                                                                                  | how the session started   | `startup`, `resume`, `clear`, `compact`                                            |
   168‚Üí| `SessionEnd`                                                                                    | why the session ended     | `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`     |
   169‚Üí| `Notification`                                                                                  | notification type         | `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`           |
   170‚Üí| `SubagentStart`                                                                                 | agent type                | `Bash`, `Explore`, `Plan`, or custom agent names                                   |
   171‚Üí| `PreCompact`                                                                                    | what triggered compaction | `manual`, `auto`                                                                   |
   172‚Üí| `SubagentStop`                                                                                  | agent type                | same values as `SubagentStart`                                                     |
   173‚Üí| `ConfigChange`                                                                                  | configuration source      | `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills` |
   174‚Üí| `UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, `WorktreeRemove` | no matcher support        | always fires on every occurrence                                                   |
   175‚Üí
   176‚ÜíThe matcher is a regex, so `Edit|Write` matches either tool and `Notebook.*` matches any tool starting with Notebook. The matcher runs against a field from the [JSON input](#hook-input-and-output) that Claude Code sends to your hook on stdin. For tool events, that field is `tool_name`. Each [hook event](#hook-events) section lists the full set of matcher values and the input schema for that event.
   177‚Üí
   178‚ÜíThis example runs a linting script only when Claude writes or edits a file:
   179‚Üí
   180‚Üí```json  theme={null}
   181‚Üí{
   182‚Üí  "hooks": {
   183‚Üí    "PostToolUse": [
   184‚Üí      {
   185‚Üí        "matcher": "Edit|Write",
   186‚Üí        "hooks": [
   187‚Üí          {
   188‚Üí            "type": "command",
   189‚Üí            "command": "/path/to/lint-check.sh"
   190‚Üí          }
   191‚Üí        ]
   192‚Üí      }
   193‚Üí    ]
   194‚Üí  }
   195‚Üí}
   196‚Üí```
   197‚Üí
   198‚Üí`UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, and `WorktreeRemove` don't support matchers and always fire on every occurrence. If you add a `matcher` field to these events, it is silently ignored.
   199‚Üí
   200‚Üí#### Match MCP tools
   201‚Üí
   202‚Üí[MCP](/en/mcp) server tools appear as regular tools in tool events (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`), so you can match them the same way you match any other tool name.
   203‚Üí
   204‚ÜíMCP tools follow the naming pattern `mcp__<server>__<tool>`, for example:
   205‚Üí
   206‚Üí* `mcp__memory__create_entities`: Memory server's create entities tool
   207‚Üí* `mcp__filesystem__read_file`: Filesystem server's read file tool
   208‚Üí* `mcp__github__search_repositories`: GitHub server's search tool
   209‚Üí
   210‚ÜíUse regex patterns to target specific MCP tools or groups of tools:
   211‚Üí
   212‚Üí* `mcp__memory__.*` matches all tools from the `memory` server
   213‚Üí* `mcp__.*__write.*` matches any tool containing "write" from any server
   214‚Üí
   215‚ÜíThis example logs all memory server operations and validates write operations from any MCP server:
   216‚Üí
   217‚Üí```json  theme={null}
   218‚Üí{
   219‚Üí  "hooks": {
   220‚Üí    "PreToolUse": [
   221‚Üí      {
   222‚Üí        "matcher": "mcp__memory__.*",
   223‚Üí        "hooks": [
   224‚Üí          {
   225‚Üí            "type": "command",
   226‚Üí            "command": "echo 'Memory operation initiated' >> ~/mcp-operations.log"
   227‚Üí          }
   228‚Üí        ]
   229‚Üí      },
   230‚Üí      {
   231‚Üí        "matcher": "mcp__.*__write.*",
   232‚Üí        "hooks": [
   233‚Üí          {
   234‚Üí            "type": "command",
   235‚Üí            "command": "/home/user/scripts/validate-mcp-write.py"
   236‚Üí          }
   237‚Üí        ]
   238‚Üí      }
   239‚Üí    ]
   240‚Üí  }
   241‚Üí}
   242‚Üí```
   243‚Üí
   244‚Üí### Hook handler fields
   245‚Üí
   246‚ÜíEach object in the inner `hooks` array is a hook handler: the shell command, HTTP endpoint, LLM prompt, or agent that runs when the matcher matches. There are four types:
   247‚Üí
   248‚Üí* **[Command hooks](#command-hook-fields)** (`type: "command"`): run a shell command. Your script receives the event's [JSON input](#hook-input-and-output) on stdin and communicates results back through exit codes and stdout.
   249‚Üí* **[HTTP hooks](#http-hook-fields)** (`type: "http"`): send the event's JSON input as an HTTP POST request to a URL. The endpoint communicates results back through the response body using the same [JSON output format](#json-output) as command hooks.
   250‚Üí* **[Prompt hooks](#prompt-and-agent-hook-fields)** (`type: "prompt"`): send a prompt to a Claude model for single-turn evaluation. The model returns a yes/no decision as JSON. See [Prompt-based hooks](#prompt-based-hooks).
   251‚Üí* **[Agent hooks](#prompt-and-agent-hook-fields)** (`type: "agent"`): spawn a subagent that can use tools like Read, Grep, and Glob to verify conditions before returning a decision. See [Agent-based hooks](#agent-based-hooks).
   252‚Üí
   253‚Üí#### Common fields
   254‚Üí
   255‚ÜíThese fields apply to all hook types:
   256‚Üí
   257‚Üí| Field           | Required | Description                                                                                                                                   |
   258‚Üí| :-------------- | :------- | :-------------------------------------------------------------------------------------------------------------------------------------------- |
   259‚Üí| `type`          | yes      | `"command"`, `"http"`, `"prompt"`, or `"agent"`                                                                                               |
   260‚Üí| `timeout`       | no       | Seconds before canceling. Defaults: 600 for command, 30 for prompt, 60 for agent                                                              |
   261‚Üí| `statusMessage` | no       | Custom spinner message displayed while the hook runs                                                                                          |
   262‚Üí| `once`          | no       | If `true`, runs only once per session then is removed. Skills only, not agents. See [Hooks in skills and agents](#hooks-in-skills-and-agents) |
   263‚Üí
   264‚Üí#### Command hook fields
   265‚Üí
   266‚ÜíIn addition to the [common fields](#common-fields), command hooks accept these fields:
   267‚Üí
   268‚Üí| Field     | Required | Description                                                                                                         |
   269‚Üí| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------ |
   270‚Üí| `command` | yes      | Shell command to execute                                                                                            |
   271‚Üí| `async`   | no       | If `true`, runs in the background without blocking. See [Run hooks in the background](#run-hooks-in-the-background) |
   272‚Üí
   273‚Üí#### HTTP hook fields
   274‚Üí
   275‚ÜíIn addition to the [common fields](#common-fields), HTTP hooks accept these fields:
   276‚Üí
   277‚Üí| Field            | Required | Description                                                                                                                                                                                      |
   278‚Üí| :--------------- | :------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
   279‚Üí| `url`            | yes      | URL to send the POST request to                                                                                                                                                                  |
   280‚Üí| `headers`        | no       | Additional HTTP headers as key-value pairs. Values support environment variable interpolation using `$VAR_NAME` or `${VAR_NAME}` syntax. Only variables listed in `allowedEnvVars` are resolved  |
   281‚Üí| `allowedEnvVars` | no       | List of environment variable names that may be interpolated into header values. References to unlisted variables are replaced with empty strings. Required for any env var interpolation to work |
   282‚Üí
   283‚ÜíClaude Code sends the hook's [JSON input](#hook-input-and-output) as the POST request body with `Content-Type: application/json`. The response body uses the same [JSON output format](#json-output) as command hooks.
   284‚Üí
   285‚ÜíError handling differs from command hooks: non-2xx responses, connection failures, and timeouts all produce non-blocking errors that allow execution to continue. To block a tool call or deny a permission, return a 2xx response with a JSON body containing `decision: "block"` or a `hookSpecificOutput` with `permissionDecision: "deny"`.
   286‚Üí
   287‚ÜíThis example sends `PreToolUse` events to a local validation service, authenticating with a token from the `MY_TOKEN` environment variable:
   288‚Üí
   289‚Üí```json  theme={null}
   290‚Üí{
   291‚Üí  "hooks": {
   292‚Üí    "PreToolUse": [
   293‚Üí      {
   294‚Üí        "matcher": "Bash",
   295‚Üí        "hooks": [
   296‚Üí          {
   297‚Üí            "type": "http",
   298‚Üí            "url": "http://localhost:8080/hooks/pre-tool-use",
   299‚Üí            "timeout": 30,
   300‚Üí            "headers": {
   301‚Üí              "Authorization": "Bearer $MY_TOKEN"
   302‚Üí            },
   303‚Üí            "allowedEnvVars": ["MY_TOKEN"]
   304‚Üí          }
   305‚Üí        ]
   306‚Üí      }
   307‚Üí    ]
   308‚Üí  }
   309‚Üí}
   310‚Üí```
   311‚Üí
   312‚Üí<Note>
   313‚Üí  HTTP hooks must be configured by editing settings JSON directly. The `/hooks` interactive menu only supports adding command hooks.
   314‚Üí</Note>
   315‚Üí
   316‚Üí#### Prompt and agent hook fields
   317‚Üí
   318‚ÜíIn addition to the [common fields](#common-fields), prompt and agent hooks accept these fields:
   319‚Üí
   320‚Üí| Field    | Required | Description                                                                                 |
   321‚Üí| :------- | :------- | :------------------------------------------------------------------------------------------ |
   322‚Üí| `prompt` | yes      | Prompt text to send to the model. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
   323‚Üí| `model`  | no       | Model to use for evaluation. Defaults to a fast model                                       |
   324‚Üí
   325‚ÜíAll matching hooks run in parallel, and identical handlers are deduplicated automatically. Command hooks are deduplicated by command string, and HTTP hooks are deduplicated by URL. Handlers run in the current directory with Claude Code's environment. The `$CLAUDE_CODE_REMOTE` environment variable is set to `"true"` in remote web environments and not set in the local CLI.
   326‚Üí
   327‚Üí### Reference scripts by path
   328‚Üí
   329‚ÜíUse environment variables to reference hook scripts relative to the project or plugin root, regardless of the working directory when the hook runs:
   330‚Üí
   331‚Üí* `$CLAUDE_PROJECT_DIR`: the project root. Wrap in quotes to handle paths with spaces.
   332‚Üí* `${CLAUDE_PLUGIN_ROOT}`: the plugin's root directory, for scripts bundled with a [plugin](/en/plugins).
   333‚Üí
   334‚Üí<Tabs>
   335‚Üí  <Tab title="Project scripts">
   336‚Üí    This example uses `$CLAUDE_PROJECT_DIR` to run a style checker from the project's `.claude/hooks/` directory after any `Write` or `Edit` tool call:
   337‚Üí
   338‚Üí    ```json  theme={null}
   339‚Üí    {
   340‚Üí      "hooks": {
   341‚Üí        "PostToolUse": [
   342‚Üí          {
   343‚Üí            "matcher": "Write|Edit",
   344‚Üí            "hooks": [
   345‚Üí              {
   346‚Üí                "type": "command",
   347‚Üí                "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/check-style.sh"
   348‚Üí              }
   349‚Üí            ]
   350‚Üí          }
   351‚Üí        ]
   352‚Üí      }
   353‚Üí    }
   354‚Üí    ```
   355‚Üí  </Tab>
   356‚Üí
   357‚Üí  <Tab title="Plugin scripts">
   358‚Üí    Define plugin hooks in `hooks/hooks.json` with an optional top-level `description` field. When a plugin is enabled, its hooks merge with your user and project hooks.
   359‚Üí
   360‚Üí    This example runs a formatting script bundled with the plugin:
   361‚Üí
   362‚Üí    ```json  theme={null}
   363‚Üí    {
   364‚Üí      "description": "Automatic code formatting",
   365‚Üí      "hooks": {
   366‚Üí        "PostToolUse": [
   367‚Üí          {
   368‚Üí            "matcher": "Write|Edit",
   369‚Üí            "hooks": [
   370‚Üí              {
   371‚Üí                "type": "command",
   372‚Üí                "command": "${CLAUDE_PLUGIN_ROOT}/scripts/format.sh",
   373‚Üí                "timeout": 30
   374‚Üí              }
   375‚Üí            ]
   376‚Üí          }
   377‚Üí        ]
   378‚Üí      }
   379‚Üí    }
   380‚Üí    ```
   381‚Üí
   382‚Üí    See the [plugin components reference](/en/plugins-reference#hooks) for details on creating plugin hooks.
   383‚Üí  </Tab>
   384‚Üí</Tabs>
   385‚Üí
   386‚Üí### Hooks in skills and agents
   387‚Üí
   388‚ÜíIn addition to settings files and plugins, hooks can be defined directly in [skills](/en/skills) and [subagents](/en/sub-agents) using frontmatter. These hooks are scoped to the component's lifecycle and only run when that component is active.
   389‚Üí
   390‚ÜíAll hook events are supported. For subagents, `Stop` hooks are automatically converted to `SubagentStop` since that is the event that fires when a subagent completes.
   391‚Üí
   392‚ÜíHooks use the same configuration format as settings-based hooks but are scoped to the component's lifetime and cleaned up when it finishes.
   393‚Üí
   394‚ÜíThis skill defines a `PreToolUse` hook that runs a security validation script before each `Bash` command:
   395‚Üí
   396‚Üí```yaml  theme={null}
   397‚Üí---
   398‚Üíname: secure-operations
   399‚Üídescription: Perform operations with security checks
   400‚Üíhooks:
   401‚Üí  PreToolUse:
   402‚Üí    - matcher: "Bash"
   403‚Üí      hooks:
   404‚Üí        - type: command
   405‚Üí          command: "./scripts/security-check.sh"
   406‚Üí---
   407‚Üí```
   408‚Üí
   409‚ÜíAgents use the same format in their YAML frontmatter.
   410‚Üí
   411‚Üí### The `/hooks` menu
   412‚Üí
   413‚ÜíType `/hooks` in Claude Code to open the interactive hooks manager, where you can view, add, and delete hooks without editing settings files directly. For a step-by-step walkthrough, see [Set up your first hook](/en/hooks-guide#set-up-your-first-hook) in the guide.
   414‚Üí
   415‚ÜíEach hook in the menu is labeled with a bracket prefix indicating its source:
   416‚Üí
   417‚Üí* `[User]`: from `~/.claude/settings.json`
   418‚Üí* `[Project]`: from `.claude/settings.json`
   419‚Üí* `[Local]`: from `.claude/settings.local.json`
   420‚Üí* `[Plugin]`: from a plugin's `hooks/hooks.json`, read-only
   421‚Üí
   422‚Üí### Disable or remove hooks
   423‚Üí
   424‚ÜíTo remove a hook, delete its entry from the settings JSON file, or use the `/hooks` menu and select the hook to delete it.
   425‚Üí
   426‚ÜíTo temporarily disable all hooks without removing them, set `"disableAllHooks": true` in your settings file or use the toggle in the `/hooks` menu. There is no way to disable an individual hook while keeping it in the configuration.
   427‚Üí
   428‚ÜíThe `disableAllHooks` setting respects the managed settings hierarchy. If an administrator has configured hooks through managed policy settings, `disableAllHooks` set in user, project, or local settings cannot disable those managed hooks. Only `disableAllHooks` set at the managed settings level can disable managed hooks.
   429‚Üí
   430‚ÜíDirect edits to hooks in settings files don't take effect immediately. Claude Code captures a snapshot of hooks at startup and uses it throughout the session. This prevents malicious or accidental hook modifications from taking effect mid-session without your review. If hooks are modified externally, Claude Code warns you and requires review in the `/hooks` menu before changes apply.
   431‚Üí
   432‚Üí## Hook input and output
   433‚Üí
   434‚ÜíCommand hooks receive JSON data via stdin and communicate results through exit codes, stdout, and stderr. HTTP hooks receive the same JSON as the POST request body and communicate results through the HTTP response body. This section covers fields and behavior common to all events. Each event's section under [Hook events](#hook-events) includes its specific input schema and decision control options.
   435‚Üí
   436‚Üí### Common input fields
   437‚Üí
   438‚ÜíAll hook events receive these fields as JSON, in addition to event-specific fields documented in each [hook event](#hook-events) section. For command hooks, this JSON arrives via stdin. For HTTP hooks, it arrives as the POST request body.
   439‚Üí
   440‚Üí| Field             | Description                                                                                                                                |
   441‚Üí| :---------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
   442‚Üí| `session_id`      | Current session identifier                                                                                                                 |
   443‚Üí| `transcript_path` | Path to conversation JSON                                                                                                                  |
   444‚Üí| `cwd`             | Current working directory when the hook is invoked                                                                                         |
   445‚Üí| `permission_mode` | Current [permission mode](/en/permissions#permission-modes): `"default"`, `"plan"`, `"acceptEdits"`, `"dontAsk"`, or `"bypassPermissions"` |
   446‚Üí| `hook_event_name` | Name of the event that fired                                                                                                               |
   447‚Üí
   448‚ÜíFor example, a `PreToolUse` hook for a Bash command receives this on stdin:
   449‚Üí
   450‚Üí```json  theme={null}
   451‚Üí{
   452‚Üí  "session_id": "abc123",
   453‚Üí  "transcript_path": "/home/user/.claude/projects/.../transcript.jsonl",
   454‚Üí  "cwd": "/home/user/my-project",
   455‚Üí  "permission_mode": "default",
   456‚Üí  "hook_event_name": "PreToolUse",
   457‚Üí  "tool_name": "Bash",
   458‚Üí  "tool_input": {
   459‚Üí    "command": "npm test"
   460‚Üí  }
   461‚Üí}
   462‚Üí```
   463‚Üí
   464‚ÜíThe `tool_name` and `tool_input` fields are event-specific. Each [hook event](#hook-events) section documents the additional fields for that event.
   465‚Üí
   466‚Üí### Exit code output
   467‚Üí
   468‚ÜíThe exit code from your hook command tells Claude Code whether the action should proceed, be blocked, or be ignored.
   469‚Üí
   470‚Üí**Exit 0** means success. Claude Code parses stdout for [JSON output fields](#json-output). JSON output is only processed on exit 0. For most events, stdout is only shown in verbose mode (`Ctrl+O`). The exceptions are `UserPromptSubmit` and `SessionStart`, where stdout is added as context that Claude can see and act on.
   471‚Üí
   472‚Üí**Exit 2** means a blocking error. Claude Code ignores stdout and any JSON in it. Instead, stderr text is fed back to Claude as an error message. The effect depends on the event: `PreToolUse` blocks the tool call, `UserPromptSubmit` rejects the prompt, and so on. See [exit code 2 behavior](#exit-code-2-behavior-per-event) for the full list.
   473‚Üí
   474‚Üí**Any other exit code** is a non-blocking error. stderr is shown in verbose mode (`Ctrl+O`) and execution continues.
   475‚Üí
   476‚ÜíFor example, a hook command script that blocks dangerous Bash commands:
   477‚Üí
   478‚Üí```bash  theme={null}
   479‚Üí#!/bin/bash
   480‚Üí# Reads JSON input from stdin, checks the command
   481‚Üícommand=$(jq -r '.tool_input.command' < /dev/stdin)
   482‚Üí
   483‚Üíif [[ "$command" == rm* ]]; then
   484‚Üí  echo "Blocked: rm commands are not allowed" >&2
   485‚Üí  exit 2  # Blocking error: tool call is prevented
   486‚Üífi
   487‚Üí
   488‚Üíexit 0  # Success: tool call proceeds
   489‚Üí```
   490‚Üí
   491‚Üí#### Exit code 2 behavior per event
   492‚Üí
   493‚ÜíExit code 2 is the way a hook signals "stop, don't do this." The effect depends on the event, because some events represent actions that can be blocked (like a tool call that hasn't happened yet) and others represent things that already happened or can't be prevented.
   494‚Üí
   495‚Üí| Hook event           | Can block? | What happens on exit 2                                                        |
   496‚Üí| :------------------- | :--------- | :---------------------------------------------------------------------------- |
   497‚Üí| `PreToolUse`         | Yes        | Blocks the tool call                                                          |
   498‚Üí| `PermissionRequest`  | Yes        | Denies the permission                                                         |
   499‚Üí| `UserPromptSubmit`   | Yes        | Blocks prompt processing and erases the prompt                                |
   500‚Üí| `Stop`               | Yes        | Prevents Claude from stopping, continues the conversation                     |
   501‚Üí| `SubagentStop`       | Yes        | Prevents the subagent from stopping                                           |
   502‚Üí| `TeammateIdle`       | Yes        | Prevents the teammate from going idle (teammate continues working)            |
   503‚Üí| `TaskCompleted`      | Yes        | Prevents the task from being marked as completed                              |
   504‚Üí| `ConfigChange`       | Yes        | Blocks the configuration change from taking effect (except `policy_settings`) |
   505‚Üí| `PostToolUse`        | No         | Shows stderr to Claude (tool already ran)                                     |
   506‚Üí| `PostToolUseFailure` | No         | Shows stderr to Claude (tool already failed)                                  |
   507‚Üí| `Notification`       | No         | Shows stderr to user only                                                     |
   508‚Üí| `SubagentStart`      | No         | Shows stderr to user only                                                     |
   509‚Üí| `SessionStart`       | No         | Shows stderr to user only                                                     |
   510‚Üí| `SessionEnd`         | No         | Shows stderr to user only                                                     |
   511‚Üí| `PreCompact`         | No         | Shows stderr to user only                                                     |
   512‚Üí| `WorktreeCreate`     | Yes        | Any non-zero exit code causes worktree creation to fail                       |
   513‚Üí| `WorktreeRemove`     | No         | Failures are logged in debug mode only                                        |
   514‚Üí
   515‚Üí### HTTP response handling
   516‚Üí
   517‚ÜíHTTP hooks use HTTP status codes and response bodies instead of exit codes and stdout:
   518‚Üí
   519‚Üí* **2xx with an empty body**: success, equivalent to exit code 0 with no output
   520‚Üí* **2xx with a plain text body**: success, the text is added as context
   521‚Üí* **2xx with a JSON body**: success, parsed using the same [JSON output](#json-output) schema as command hooks
   522‚Üí* **Non-2xx status**: non-blocking error, execution continues
   523‚Üí* **Connection failure or timeout**: non-blocking error, execution continues
   524‚Üí
   525‚ÜíUnlike command hooks, HTTP hooks cannot signal a blocking error through status codes alone. To block a tool call or deny a permission, return a 2xx response with a JSON body containing the appropriate decision fields.
   526‚Üí
   527‚Üí### JSON output
   528‚Üí
   529‚ÜíExit codes let you allow or block, but JSON output gives you finer-grained control. Instead of exiting with code 2 to block, exit 0 and print a JSON object to stdout. Claude Code reads specific fields from that JSON to control behavior, including [decision control](#decision-control) for blocking, allowing, or escalating to the user.
   530‚Üí
   531‚Üí<Note>
   532‚Üí  You must choose one approach per hook, not both: either use exit codes alone for signaling, or exit 0 and print JSON for structured control. Claude Code only processes JSON on exit 0. If you exit 2, any JSON is ignored.
   533‚Üí</Note>
   534‚Üí
   535‚ÜíYour hook's stdout must contain only the JSON object. If your shell profile prints text on startup, it can interfere with JSON parsing. See [JSON validation failed](/en/hooks-guide#json-validation-failed) in the troubleshooting guide.
   536‚Üí
   537‚ÜíThe JSON object supports three kinds of fields:
   538‚Üí
   539‚Üí* **Universal fields** like `continue` work across all events. These are listed in the table below.
   540‚Üí* **Top-level `decision` and `reason`** are used by some events to block or provide feedback.
   541‚Üí* **`hookSpecificOutput`** is a nested object for events that need richer control. It requires a `hookEventName` field set to the event name.
   542‚Üí
   543‚Üí| Field            | Default | Description                                                                                                                |
   544‚Üí| :--------------- | :------ | :------------------------------------------------------------------------------------------------------------------------- |
   545‚Üí| `continue`       | `true`  | If `false`, Claude stops processing entirely after the hook runs. Takes precedence over any event-specific decision fields |
   546‚Üí| `stopReason`     | none    | Message shown to the user when `continue` is `false`. Not shown to Claude                                                  |
   547‚Üí| `suppressOutput` | `false` | If `true`, hides stdout from verbose mode output                                                                           |
   548‚Üí| `systemMessage`  | none    | Warning message shown to the user                                                                                          |
   549‚Üí
   550‚ÜíTo stop Claude entirely regardless of event type:
   551‚Üí
   552‚Üí```json  theme={null}
   553‚Üí{ "continue": false, "stopReason": "Build failed, fix errors before continuing" }
   554‚Üí```
   555‚Üí
   556‚Üí#### Decision control
   557‚Üí
   558‚ÜíNot every event supports blocking or controlling behavior through JSON. The events that do each use a different set of fields to express that decision. Use this table as a quick reference before writing a hook:
   559‚Üí
   560‚Üí| Events                                                                              | Decision pattern     | Key fields                                                                  |
   561‚Üí| :---------------------------------------------------------------------------------- | :------------------- | :-------------------------------------------------------------------------- |
   562‚Üí| UserPromptSubmit, PostToolUse, PostToolUseFailure, Stop, SubagentStop, ConfigChange | Top-level `decision` | `decision: "block"`, `reason`                                               |
   563‚Üí| TeammateIdle, TaskCompleted                                                         | Exit code only       | Exit code 2 blocks the action, stderr is fed back as feedback               |
   564‚Üí| PreToolUse                                                                          | `hookSpecificOutput` | `permissionDecision` (allow/deny/ask), `permissionDecisionReason`           |
   565‚Üí| PermissionRequest                                                                   | `hookSpecificOutput` | `decision.behavior` (allow/deny)                                            |
   566‚Üí| WorktreeCreate                                                                      | stdout path          | Hook prints absolute path to created worktree. Non-zero exit fails creation |
   567‚Üí| WorktreeRemove, Notification, SessionEnd, PreCompact                                | None                 | No decision control. Used for side effects like logging or cleanup          |
   568‚Üí
   569‚ÜíHere are examples of each pattern in action:
   570‚Üí
   571‚Üí<Tabs>
   572‚Üí  <Tab title="Top-level decision">
   573‚Üí    Used by `UserPromptSubmit`, `PostToolUse`, `PostToolUseFailure`, `Stop`, `SubagentStop`, and `ConfigChange`. The only value is `"block"`. To allow the action to proceed, omit `decision` from your JSON, or exit 0 without any JSON at all:
   574‚Üí
   575‚Üí    ```json  theme={null}
   576‚Üí    {
   577‚Üí      "decision": "block",
   578‚Üí      "reason": "Test suite must pass before proceeding"
   579‚Üí    }
   580‚Üí    ```
   581‚Üí  </Tab>
   582‚Üí
   583‚Üí  <Tab title="PreToolUse">
   584‚Üí    Uses `hookSpecificOutput` for richer control: allow, deny, or escalate to the user. You can also modify tool input before it runs or inject additional context for Claude. See [PreToolUse decision control](#pretooluse-decision-control) for the full set of options.
   585‚Üí
   586‚Üí    ```json  theme={null}
   587‚Üí    {
   588‚Üí      "hookSpecificOutput": {
   589‚Üí        "hookEventName": "PreToolUse",
   590‚Üí        "permissionDecision": "deny",
   591‚Üí        "permissionDecisionReason": "Database writes are not allowed"
   592‚Üí      }
   593‚Üí    }
   594‚Üí    ```
   595‚Üí  </Tab>
   596‚Üí
   597‚Üí  <Tab title="PermissionRequest">
   598‚Üí    Uses `hookSpecificOutput` to allow or deny a permission request on behalf of the user. When allowing, you can also modify the tool's input or apply permission rules so the user isn't prompted again. See [PermissionRequest decision control](#permissionrequest-decision-control) for the full set of options.
   599‚Üí
   600‚Üí    ```json  theme={null}
   601‚Üí    {
   602‚Üí      "hookSpecificOutput": {
   603‚Üí        "hookEventName": "PermissionRequest",
   604‚Üí        "decision": {
   605‚Üí          "behavior": "allow",
   606‚Üí          "updatedInput": {
   607‚Üí            "command": "npm run lint"
   608‚Üí          }
   609‚Üí        }
   610‚Üí      }
   611‚Üí    }
   612‚Üí    ```
   613‚Üí  </Tab>
   614‚Üí</Tabs>
   615‚Üí
   616‚ÜíFor extended examples including Bash command validation, prompt filtering, and auto-approval scripts, see [What you can automate](/en/hooks-guide#what-you-can-automate) in the guide and the [Bash command validator reference implementation](https://github.com/anthropics/claude-code/blob/main/examples/hooks/bash_command_validator_example.py).
   617‚Üí
   618‚Üí## Hook events
   619‚Üí
   620‚ÜíEach event corresponds to a point in Claude Code's lifecycle where hooks can run. The sections below are ordered to match the lifecycle: from session setup through the agentic loop to session end. Each section describes when the event fires, what matchers it supports, the JSON input it receives, and how to control behavior through output.
   621‚Üí
   622‚Üí### SessionStart
   623‚Üí
   624‚ÜíRuns when Claude Code starts a new session or resumes an existing session. Useful for loading development context like existing issues or recent changes to your codebase, or setting up environment variables. For static context that does not require a script, use [CLAUDE.md](/en/memory) instead.
   625‚Üí
   626‚ÜíSessionStart runs on every session, so keep these hooks fast.
   627‚Üí
   628‚ÜíThe matcher value corresponds to how the session was initiated:
   629‚Üí
   630‚Üí| Matcher   | When it fires                          |
   631‚Üí| :-------- | :------------------------------------- |
   632‚Üí| `startup` | New session                            |
   633‚Üí| `resume`  | `--resume`, `--continue`, or `/resume` |
   634‚Üí| `clear`   | `/clear`                               |
   635‚Üí| `compact` | Auto or manual compaction              |
   636‚Üí
   637‚Üí#### SessionStart input
   638‚Üí
   639‚ÜíIn addition to the [common input fields](#common-input-fields), SessionStart hooks receive `source`, `model`, and optionally `agent_type`. The `source` field indicates how the session started: `"startup"` for new sessions, `"resume"` for resumed sessions, `"clear"` after `/clear`, or `"compact"` after compaction. The `model` field contains the model identifier. If you start Claude Code with `claude --agent <name>`, an `agent_type` field contains the agent name.
   640‚Üí
   641‚Üí```json  theme={null}
   642‚Üí{
   643‚Üí  "session_id": "abc123",
   644‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   645‚Üí  "cwd": "/Users/...",
   646‚Üí  "permission_mode": "default",
   647‚Üí  "hook_event_name": "SessionStart",
   648‚Üí  "source": "startup",
   649‚Üí  "model": "claude-sonnet-4-6"
   650‚Üí}
   651‚Üí```
   652‚Üí
   653‚Üí#### SessionStart decision control
   654‚Üí
   655‚ÜíAny text your hook script prints to stdout is added as context for Claude. In addition to the [JSON output fields](#json-output) available to all hooks, you can return these event-specific fields:
   656‚Üí
   657‚Üí| Field               | Description                                                               |
   658‚Üí| :------------------ | :------------------------------------------------------------------------ |
   659‚Üí| `additionalContext` | String added to Claude's context. Multiple hooks' values are concatenated |
   660‚Üí
   661‚Üí```json  theme={null}
   662‚Üí{
   663‚Üí  "hookSpecificOutput": {
   664‚Üí    "hookEventName": "SessionStart",
   665‚Üí    "additionalContext": "My additional context here"
   666‚Üí  }
   667‚Üí}
   668‚Üí```
   669‚Üí
   670‚Üí#### Persist environment variables
   671‚Üí
   672‚ÜíSessionStart hooks have access to the `CLAUDE_ENV_FILE` environment variable, which provides a file path where you can persist environment variables for subsequent Bash commands.
   673‚Üí
   674‚ÜíTo set individual environment variables, write `export` statements to `CLAUDE_ENV_FILE`. Use append (`>>`) to preserve variables set by other hooks:
   675‚Üí
   676‚Üí```bash  theme={null}
   677‚Üí#!/bin/bash
   678‚Üí
   679‚Üíif [ -n "$CLAUDE_ENV_FILE" ]; then
   680‚Üí  echo 'export NODE_ENV=production' >> "$CLAUDE_ENV_FILE"
   681‚Üí  echo 'export DEBUG_LOG=true' >> "$CLAUDE_ENV_FILE"
   682‚Üí  echo 'export PATH="$PATH:./node_modules/.bin"' >> "$CLAUDE_ENV_FILE"
   683‚Üífi
   684‚Üí
   685‚Üíexit 0
   686‚Üí```
   687‚Üí
   688‚ÜíTo capture all environment changes from setup commands, compare the exported variables before and after:
   689‚Üí
   690‚Üí```bash  theme={null}
   691‚Üí#!/bin/bash
   692‚Üí
   693‚ÜíENV_BEFORE=$(export -p | sort)
   694‚Üí
   695‚Üí# Run your setup commands that modify the environment
   696‚Üísource ~/.nvm/nvm.sh
   697‚Üínvm use 20
   698‚Üí
   699‚Üíif [ -n "$CLAUDE_ENV_FILE" ]; then
   700‚Üí  ENV_AFTER=$(export -p | sort)
   701‚Üí  comm -13 <(echo "$ENV_BEFORE") <(echo "$ENV_AFTER") >> "$CLAUDE_ENV_FILE"
   702‚Üífi
   703‚Üí
   704‚Üíexit 0
   705‚Üí```
   706‚Üí
   707‚ÜíAny variables written to this file will be available in all subsequent Bash commands that Claude Code executes during the session.
   708‚Üí
   709‚Üí<Note>
   710‚Üí  `CLAUDE_ENV_FILE` is available for SessionStart hooks. Other hook types do not have access to this variable.
   711‚Üí</Note>
   712‚Üí
   713‚Üí### UserPromptSubmit
   714‚Üí
   715‚ÜíRuns when the user submits a prompt, before Claude processes it. This allows you
   716‚Üíto add additional context based on the prompt/conversation, validate prompts, or
   717‚Üíblock certain types of prompts.
   718‚Üí
   719‚Üí#### UserPromptSubmit input
   720‚Üí
   721‚ÜíIn addition to the [common input fields](#common-input-fields), UserPromptSubmit hooks receive the `prompt` field containing the text the user submitted.
   722‚Üí
   723‚Üí```json  theme={null}
   724‚Üí{
   725‚Üí  "session_id": "abc123",
   726‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   727‚Üí  "cwd": "/Users/...",
   728‚Üí  "permission_mode": "default",
   729‚Üí  "hook_event_name": "UserPromptSubmit",
   730‚Üí  "prompt": "Write a function to calculate the factorial of a number"
   731‚Üí}
   732‚Üí```
   733‚Üí
   734‚Üí#### UserPromptSubmit decision control
   735‚Üí
   736‚Üí`UserPromptSubmit` hooks can control whether a user prompt is processed and add context. All [JSON output fields](#json-output) are available.
   737‚Üí
   738‚ÜíThere are two ways to add context to the conversation on exit code 0:
   739‚Üí
   740‚Üí* **Plain text stdout**: any non-JSON text written to stdout is added as context
   741‚Üí* **JSON with `additionalContext`**: use the JSON format below for more control. The `additionalContext` field is added as context
   742‚Üí
   743‚ÜíPlain stdout is shown as hook output in the transcript. The `additionalContext` field is added more discretely.
   744‚Üí
   745‚ÜíTo block a prompt, return a JSON object with `decision` set to `"block"`:
   746‚Üí
   747‚Üí| Field               | Description                                                                                                        |
   748‚Üí| :------------------ | :----------------------------------------------------------------------------------------------------------------- |
   749‚Üí| `decision`          | `"block"` prevents the prompt from being processed and erases it from context. Omit to allow the prompt to proceed |
   750‚Üí| `reason`            | Shown to the user when `decision` is `"block"`. Not added to context                                               |
   751‚Üí| `additionalContext` | String added to Claude's context                                                                                   |
   752‚Üí
   753‚Üí```json  theme={null}
   754‚Üí{
   755‚Üí  "decision": "block",
   756‚Üí  "reason": "Explanation for decision",
   757‚Üí  "hookSpecificOutput": {
   758‚Üí    "hookEventName": "UserPromptSubmit",
   759‚Üí    "additionalContext": "My additional context here"
   760‚Üí  }
   761‚Üí}
   762‚Üí```
   763‚Üí
   764‚Üí<Note>
   765‚Üí  The JSON format isn't required for simple use cases. To add context, you can print plain text to stdout with exit code 0. Use JSON when you need to
   766‚Üí  block prompts or want more structured control.
   767‚Üí</Note>
   768‚Üí
   769‚Üí### PreToolUse
   770‚Üí
   771‚ÜíRuns after Claude creates tool parameters and before processing the tool call. Matches on tool name: `Bash`, `Edit`, `Write`, `Read`, `Glob`, `Grep`, `Agent`, `WebFetch`, `WebSearch`, and any [MCP tool names](#match-mcp-tools).
   772‚Üí
   773‚ÜíUse [PreToolUse decision control](#pretooluse-decision-control) to allow, deny, or ask for permission to use the tool.
   774‚Üí
   775‚Üí#### PreToolUse input
   776‚Üí
   777‚ÜíIn addition to the [common input fields](#common-input-fields), PreToolUse hooks receive `tool_name`, `tool_input`, and `tool_use_id`. The `tool_input` fields depend on the tool:
   778‚Üí
   779‚Üí##### Bash
   780‚Üí
   781‚ÜíExecutes shell commands.
   782‚Üí
   783‚Üí| Field               | Type    | Example            | Description                                   |
   784‚Üí| :------------------ | :------ | :----------------- | :-------------------------------------------- |
   785‚Üí| `command`           | string  | `"npm test"`       | The shell command to execute                  |
   786‚Üí| `description`       | string  | `"Run test suite"` | Optional description of what the command does |
   787‚Üí| `timeout`           | number  | `120000`           | Optional timeout in milliseconds              |
   788‚Üí| `run_in_background` | boolean | `false`            | Whether to run the command in background      |
   789‚Üí
   790‚Üí##### Write
   791‚Üí
   792‚ÜíCreates or overwrites a file.
   793‚Üí
   794‚Üí| Field       | Type   | Example               | Description                        |
   795‚Üí| :---------- | :----- | :-------------------- | :--------------------------------- |
   796‚Üí| `file_path` | string | `"/path/to/file.txt"` | Absolute path to the file to write |
   797‚Üí| `content`   | string | `"file content"`      | Content to write to the file       |
   798‚Üí
   799‚Üí##### Edit
   800‚Üí
   801‚ÜíReplaces a string in an existing file.
   802‚Üí
   803‚Üí| Field         | Type    | Example               | Description                        |
   804‚Üí| :------------ | :------ | :-------------------- | :--------------------------------- |
   805‚Üí| `file_path`   | string  | `"/path/to/file.txt"` | Absolute path to the file to edit  |
   806‚Üí| `old_string`  | string  | `"original text"`     | Text to find and replace           |
   807‚Üí| `new_string`  | string  | `"replacement text"`  | Replacement text                   |
   808‚Üí| `replace_all` | boolean | `false`               | Whether to replace all occurrences |
   809‚Üí
   810‚Üí##### Read
   811‚Üí
   812‚ÜíReads file contents.
   813‚Üí
   814‚Üí| Field       | Type   | Example               | Description                                |
   815‚Üí| :---------- | :----- | :-------------------- | :----------------------------------------- |
   816‚Üí| `file_path` | string | `"/path/to/file.txt"` | Absolute path to the file to read          |
   817‚Üí| `offset`    | number | `10`                  | Optional line number to start reading from |
   818‚Üí| `limit`     | number | `50`                  | Optional number of lines to read           |
   819‚Üí
   820‚Üí##### Glob
   821‚Üí
   822‚ÜíFinds files matching a glob pattern.
   823‚Üí
   824‚Üí| Field     | Type   | Example          | Description                                                            |
   825‚Üí| :-------- | :----- | :--------------- | :--------------------------------------------------------------------- |
   826‚Üí| `pattern` | string | `"**/*.ts"`      | Glob pattern to match files against                                    |
   827‚Üí| `path`    | string | `"/path/to/dir"` | Optional directory to search in. Defaults to current working directory |
   828‚Üí
   829‚Üí##### Grep
   830‚Üí
   831‚ÜíSearches file contents with regular expressions.
   832‚Üí
   833‚Üí| Field         | Type    | Example          | Description                                                                           |
   834‚Üí| :------------ | :------ | :--------------- | :------------------------------------------------------------------------------------ |
   835‚Üí| `pattern`     | string  | `"TODO.*fix"`    | Regular expression pattern to search for                                              |
   836‚Üí| `path`        | string  | `"/path/to/dir"` | Optional file or directory to search in                                               |
   837‚Üí| `glob`        | string  | `"*.ts"`         | Optional glob pattern to filter files                                                 |
   838‚Üí| `output_mode` | string  | `"content"`      | `"content"`, `"files_with_matches"`, or `"count"`. Defaults to `"files_with_matches"` |
   839‚Üí| `-i`          | boolean | `true`           | Case insensitive search                                                               |
   840‚Üí| `multiline`   | boolean | `false`          | Enable multiline matching                                                             |
   841‚Üí
   842‚Üí##### WebFetch
   843‚Üí
   844‚ÜíFetches and processes web content.
   845‚Üí
   846‚Üí| Field    | Type   | Example                       | Description                          |
   847‚Üí| :------- | :----- | :---------------------------- | :----------------------------------- |
   848‚Üí| `url`    | string | `"https://example.com/api"`   | URL to fetch content from            |
   849‚Üí| `prompt` | string | `"Extract the API endpoints"` | Prompt to run on the fetched content |
   850‚Üí
   851‚Üí##### WebSearch
   852‚Üí
   853‚ÜíSearches the web.
   854‚Üí
   855‚Üí| Field             | Type   | Example                        | Description                                       |
   856‚Üí| :---------------- | :----- | :----------------------------- | :------------------------------------------------ |
   857‚Üí| `query`           | string | `"react hooks best practices"` | Search query                                      |
   858‚Üí| `allowed_domains` | array  | `["docs.example.com"]`         | Optional: only include results from these domains |
   859‚Üí| `blocked_domains` | array  | `["spam.example.com"]`         | Optional: exclude results from these domains      |
   860‚Üí
   861‚Üí##### Agent
   862‚Üí
   863‚ÜíSpawns a [subagent](/en/sub-agents).
   864‚Üí
   865‚Üí| Field           | Type   | Example                    | Description                                  |
   866‚Üí| :-------------- | :----- | :------------------------- | :------------------------------------------- |
   867‚Üí| `prompt`        | string | `"Find all API endpoints"` | The task for the agent to perform            |
   868‚Üí| `description`   | string | `"Find API endpoints"`     | Short description of the task                |
   869‚Üí| `subagent_type` | string | `"Explore"`                | Type of specialized agent to use             |
   870‚Üí| `model`         | string | `"sonnet"`                 | Optional model alias to override the default |
   871‚Üí
   872‚Üí#### PreToolUse decision control
   873‚Üí
   874‚Üí`PreToolUse` hooks can control whether a tool call proceeds. Unlike other hooks that use a top-level `decision` field, PreToolUse returns its decision inside a `hookSpecificOutput` object. This gives it richer control: three outcomes (allow, deny, or ask) plus the ability to modify tool input before execution.
   875‚Üí
   876‚Üí| Field                      | Description                                                                                                                                      |
   877‚Üí| :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------- |
   878‚Üí| `permissionDecision`       | `"allow"` bypasses the permission system, `"deny"` prevents the tool call, `"ask"` prompts the user to confirm                                   |
   879‚Üí| `permissionDecisionReason` | For `"allow"` and `"ask"`, shown to the user but not Claude. For `"deny"`, shown to Claude                                                       |
   880‚Üí| `updatedInput`             | Modifies the tool's input parameters before execution. Combine with `"allow"` to auto-approve, or `"ask"` to show the modified input to the user |
   881‚Üí| `additionalContext`        | String added to Claude's context before the tool executes                                                                                        |
   882‚Üí
   883‚Üí```json  theme={null}
   884‚Üí{
   885‚Üí  "hookSpecificOutput": {
   886‚Üí    "hookEventName": "PreToolUse",
   887‚Üí    "permissionDecision": "allow",
   888‚Üí    "permissionDecisionReason": "My reason here",
   889‚Üí    "updatedInput": {
   890‚Üí      "field_to_modify": "new value"
   891‚Üí    },
   892‚Üí    "additionalContext": "Current environment: production. Proceed with caution."
   893‚Üí  }
   894‚Üí}
   895‚Üí```
   896‚Üí
   897‚Üí<Note>
   898‚Üí  PreToolUse previously used top-level `decision` and `reason` fields, but these are deprecated for this event. Use `hookSpecificOutput.permissionDecision` and `hookSpecificOutput.permissionDecisionReason` instead. The deprecated values `"approve"` and `"block"` map to `"allow"` and `"deny"` respectively. Other events like PostToolUse and Stop continue to use top-level `decision` and `reason` as their current format.
   899‚Üí</Note>
   900‚Üí
   901‚Üí### PermissionRequest
   902‚Üí
   903‚ÜíRuns when the user is shown a permission dialog.
   904‚ÜíUse [PermissionRequest decision control](#permissionrequest-decision-control) to allow or deny on behalf of the user.
   905‚Üí
   906‚ÜíMatches on tool name, same values as PreToolUse.
   907‚Üí
   908‚Üí#### PermissionRequest input
   909‚Üí
   910‚ÜíPermissionRequest hooks receive `tool_name` and `tool_input` fields like PreToolUse hooks, but without `tool_use_id`. An optional `permission_suggestions` array contains the "always allow" options the user would normally see in the permission dialog. The difference is when the hook fires: PermissionRequest hooks run when a permission dialog is about to be shown to the user, while PreToolUse hooks run before tool execution regardless of permission status.
   911‚Üí
   912‚Üí```json  theme={null}
   913‚Üí{
   914‚Üí  "session_id": "abc123",
   915‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   916‚Üí  "cwd": "/Users/...",
   917‚Üí  "permission_mode": "default",
   918‚Üí  "hook_event_name": "PermissionRequest",
   919‚Üí  "tool_name": "Bash",
   920‚Üí  "tool_input": {
   921‚Üí    "command": "rm -rf node_modules",
   922‚Üí    "description": "Remove node_modules directory"
   923‚Üí  },
   924‚Üí  "permission_suggestions": [
   925‚Üí    { "type": "toolAlwaysAllow", "tool": "Bash" }
   926‚Üí  ]
   927‚Üí}
   928‚Üí```
   929‚Üí
   930‚Üí#### PermissionRequest decision control
   931‚Üí
   932‚Üí`PermissionRequest` hooks can allow or deny permission requests. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return a `decision` object with these event-specific fields:
   933‚Üí
   934‚Üí| Field                | Description                                                                                                    |
   935‚Üí| :------------------- | :------------------------------------------------------------------------------------------------------------- |
   936‚Üí| `behavior`           | `"allow"` grants the permission, `"deny"` denies it                                                            |
   937‚Üí| `updatedInput`       | For `"allow"` only: modifies the tool's input parameters before execution                                      |
   938‚Üí| `updatedPermissions` | For `"allow"` only: applies permission rule updates, equivalent to the user selecting an "always allow" option |
   939‚Üí| `message`            | For `"deny"` only: tells Claude why the permission was denied                                                  |
   940‚Üí| `interrupt`          | For `"deny"` only: if `true`, stops Claude                                                                     |
   941‚Üí
   942‚Üí```json  theme={null}
   943‚Üí{
   944‚Üí  "hookSpecificOutput": {
   945‚Üí    "hookEventName": "PermissionRequest",
   946‚Üí    "decision": {
   947‚Üí      "behavior": "allow",
   948‚Üí      "updatedInput": {
   949‚Üí        "command": "npm run lint"
   950‚Üí      }
   951‚Üí    }
   952‚Üí  }
   953‚Üí}
   954‚Üí```
   955‚Üí
   956‚Üí### PostToolUse
   957‚Üí
   958‚ÜíRuns immediately after a tool completes successfully.
   959‚Üí
   960‚ÜíMatches on tool name, same values as PreToolUse.
   961‚Üí
   962‚Üí#### PostToolUse input
   963‚Üí
   964‚Üí`PostToolUse` hooks fire after a tool has already executed successfully. The input includes both `tool_input`, the arguments sent to the tool, and `tool_response`, the result it returned. The exact schema for both depends on the tool.
   965‚Üí
   966‚Üí```json  theme={null}
   967‚Üí{
   968‚Üí  "session_id": "abc123",
   969‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   970‚Üí  "cwd": "/Users/...",
   971‚Üí  "permission_mode": "default",
   972‚Üí  "hook_event_name": "PostToolUse",
   973‚Üí  "tool_name": "Write",
   974‚Üí  "tool_input": {
   975‚Üí    "file_path": "/path/to/file.txt",
   976‚Üí    "content": "file content"
   977‚Üí  },
   978‚Üí  "tool_response": {
   979‚Üí    "filePath": "/path/to/file.txt",
   980‚Üí    "success": true
   981‚Üí  },
   982‚Üí  "tool_use_id": "toolu_01ABC123..."
   983‚Üí}
   984‚Üí```
   985‚Üí
   986‚Üí#### PostToolUse decision control
   987‚Üí
   988‚Üí`PostToolUse` hooks can provide feedback to Claude after tool execution. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
   989‚Üí
   990‚Üí| Field                  | Description                                                                                |
   991‚Üí| :--------------------- | :----------------------------------------------------------------------------------------- |
   992‚Üí| `decision`             | `"block"` prompts Claude with the `reason`. Omit to allow the action to proceed            |
   993‚Üí| `reason`               | Explanation shown to Claude when `decision` is `"block"`                                   |
   994‚Üí| `additionalContext`    | Additional context for Claude to consider                                                  |
   995‚Üí| `updatedMCPToolOutput` | For [MCP tools](#match-mcp-tools) only: replaces the tool's output with the provided value |
   996‚Üí
   997‚Üí```json  theme={null}
   998‚Üí{
   999‚Üí  "decision": "block",
  1000‚Üí  "reason": "Explanation for decision",
  1001‚Üí  "hookSpecificOutput": {
  1002‚Üí    "hookEventName": "PostToolUse",
  1003‚Üí    "additionalContext": "Additional information for Claude"
  1004‚Üí  }
  1005‚Üí}
  1006‚Üí```
  1007‚Üí
  1008‚Üí### PostToolUseFailure
  1009‚Üí
  1010‚ÜíRuns when a tool execution fails. This event fires for tool calls that throw errors or return failure results. Use this to log failures, send alerts, or provide corrective feedback to Claude.
  1011‚Üí
  1012‚ÜíMatches on tool name, same values as PreToolUse.
  1013‚Üí
  1014‚Üí#### PostToolUseFailure input
  1015‚Üí
  1016‚ÜíPostToolUseFailure hooks receive the same `tool_name` and `tool_input` fields as PostToolUse, along with error information as top-level fields:
  1017‚Üí
  1018‚Üí```json  theme={null}
  1019‚Üí{
  1020‚Üí  "session_id": "abc123",
  1021‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1022‚Üí  "cwd": "/Users/...",
  1023‚Üí  "permission_mode": "default",
  1024‚Üí  "hook_event_name": "PostToolUseFailure",
  1025‚Üí  "tool_name": "Bash",
  1026‚Üí  "tool_input": {
  1027‚Üí    "command": "npm test",
  1028‚Üí    "description": "Run test suite"
  1029‚Üí  },
  1030‚Üí  "tool_use_id": "toolu_01ABC123...",
  1031‚Üí  "error": "Command exited with non-zero status code 1",
  1032‚Üí  "is_interrupt": false
  1033‚Üí}
  1034‚Üí```
  1035‚Üí
  1036‚Üí| Field          | Description                                                                     |
  1037‚Üí| :------------- | :------------------------------------------------------------------------------ |
  1038‚Üí| `error`        | String describing what went wrong                                               |
  1039‚Üí| `is_interrupt` | Optional boolean indicating whether the failure was caused by user interruption |
  1040‚Üí
  1041‚Üí#### PostToolUseFailure decision control
  1042‚Üí
  1043‚Üí`PostToolUseFailure` hooks can provide context to Claude after a tool failure. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
  1044‚Üí
  1045‚Üí| Field               | Description                                                   |
  1046‚Üí| :------------------ | :------------------------------------------------------------ |
  1047‚Üí| `additionalContext` | Additional context for Claude to consider alongside the error |
  1048‚Üí
  1049‚Üí```json  theme={null}
  1050‚Üí{
  1051‚Üí  "hookSpecificOutput": {
  1052‚Üí    "hookEventName": "PostToolUseFailure",
  1053‚Üí    "additionalContext": "Additional information about the failure for Claude"
  1054‚Üí  }
  1055‚Üí}
  1056‚Üí```
  1057‚Üí
  1058‚Üí### Notification
  1059‚Üí
  1060‚ÜíRuns when Claude Code sends notifications. Matches on notification type: `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`. Omit the matcher to run hooks for all notification types.
  1061‚Üí
  1062‚ÜíUse separate matchers to run different handlers depending on the notification type. This configuration triggers a permission-specific alert script when Claude needs permission approval and a different notification when Claude has been idle:
  1063‚Üí
  1064‚Üí```json  theme={null}
  1065‚Üí{
  1066‚Üí  "hooks": {
  1067‚Üí    "Notification": [
  1068‚Üí      {
  1069‚Üí        "matcher": "permission_prompt",
  1070‚Üí        "hooks": [
  1071‚Üí          {
  1072‚Üí            "type": "command",
  1073‚Üí            "command": "/path/to/permission-alert.sh"
  1074‚Üí          }
  1075‚Üí        ]
  1076‚Üí      },
  1077‚Üí      {
  1078‚Üí        "matcher": "idle_prompt",
  1079‚Üí        "hooks": [
  1080‚Üí          {
  1081‚Üí            "type": "command",
  1082‚Üí            "command": "/path/to/idle-notification.sh"
  1083‚Üí          }
  1084‚Üí        ]
  1085‚Üí      }
  1086‚Üí    ]
  1087‚Üí  }
  1088‚Üí}
  1089‚Üí```
  1090‚Üí
  1091‚Üí#### Notification input
  1092‚Üí
  1093‚ÜíIn addition to the [common input fields](#common-input-fields), Notification hooks receive `message` with the notification text, an optional `title`, and `notification_type` indicating which type fired.
  1094‚Üí
  1095‚Üí```json  theme={null}
  1096‚Üí{
  1097‚Üí  "session_id": "abc123",
  1098‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1099‚Üí  "cwd": "/Users/...",

  1174‚Üí### Stop
  1175‚Üí
  1176‚ÜíRuns when the main Claude Code agent has finished responding. Does not run if
  1177‚Üíthe stoppage occurred due to a user interrupt.

  1182‚Üí
  1183‚Üí```json  theme={null}
  1184‚Üí{

  1189‚Üí  "hook_event_name": "Stop",

  1198‚Üí
  1199‚Üí| Field      | Description                                                                |
  1200‚Üí| :--------- | :------------------------------------------------------------------------- |
  1201‚Üí| `decision` | `"block"` prevents Claude from stopping. Omit to allow Claude to stop      |

  1233‚Üí| Field           | Description                                   |
  1234‚Üí| :-------------- | :-------------------------------------------- |

  1268‚Üí  "permission_mode": "default",
  1269‚Üí  "hook_event_name": "TaskCompleted",
  1270‚Üí  "task_id": "task-001",
  1271‚Üí  "task_subject": "Implement user authentication",
  1272‚Üí  "task_description": "Add login and signup endpoints",

  1274‚Üí  "team_name": "my-project"

  1278‚Üí| Field              | Description                                             |
  1279‚Üí| :----------------- | :------------------------------------------------------ |
  1280‚Üí| `task_id`          | Identifier of the task being completed                  |

  1282‚Üí| `task_description` | Detailed description of the task. May be absent         |

  1286‚Üí#### TaskCompleted decision control
  1287‚Üí
  1288‚ÜíTaskCompleted hooks use exit codes only, not JSON decision control. This example runs tests and blocks task completion if they fail:

  1304‚Üí### ConfigChange
  1305‚Üí
  1306‚ÜíRuns when a configuration file changes during a session. Use this to audit settings changes, enforce security policies, or block unauthorized modifications to configuration files.
  1307‚Üí
  1308‚ÜíConfigChange hooks fire for changes to settings files, managed policy settings, and skill files. The `source` field in the input tells you which type of configuration changed, and the optional `file_path` field provides the path to the changed file.

  1310‚ÜíThe matcher filters on the configuration source:
  1311‚Üí
  1312‚Üí| Matcher            | When it fires                             |
  1313‚Üí| :----------------- | :---------------------------------------- |
  1314‚Üí| `user_settings`    | `~/.claude/settings.json` changes         |
  1315‚Üí| `project_settings` | `.claude/settings.json` changes           |
  1316‚Üí| `local_settings`   | `.claude/settings.local.json` changes     |
  1317‚Üí| `policy_settings`  | Managed policy settings change            |
  1318‚Üí| `skills`           | A skill file in `.claude/skills/` changes |
  1319‚Üí
  1320‚ÜíThis example logs all configuration changes for security auditing:
  1321‚Üí
  1322‚Üí```json  theme={null}
  1323‚Üí{
  1324‚Üí  "hooks": {
  1325‚Üí    "ConfigChange": [
  1326‚Üí      {
  1327‚Üí        "hooks": [
  1328‚Üí          {
  1329‚Üí            "type": "command",
  1330‚Üí            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/audit-config-change.sh"
  1331‚Üí          }
  1332‚Üí        ]
  1333‚Üí      }
  1334‚Üí    ]
  1335‚Üí  }
  1336‚Üí}
  1337‚Üí```
  1338‚Üí
  1339‚Üí#### ConfigChange input
  1340‚Üí
  1341‚ÜíIn addition to the [common input fields](#common-input-fields), ConfigChange hooks receive `source` and optionally `file_path`. The `source` field indicates which configuration type changed, and `file_path` provides the path to the specific file that was modified.
  1342‚Üí
  1343‚Üí```json  theme={null}
  1344‚Üí{
  1345‚Üí  "session_id": "abc123",
  1346‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1347‚Üí  "cwd": "/Users/...",
  1348‚Üí  "permission_mode": "default",
  1349‚Üí  "hook_event_name": "ConfigChange",
  1350‚Üí  "source": "project_settings",
  1351‚Üí  "file_path": "/Users/.../my-project/.claude/settings.json"
  1352‚Üí}
  1353‚Üí```
  1354‚Üí
  1355‚Üí#### ConfigChange decision control
  1356‚Üí
  1357‚ÜíConfigChange hooks can block configuration changes from taking effect. Use exit code 2 or a JSON `decision` to prevent the change. When blocked, the new settings are not applied to the running session.
  1358‚Üí
  1359‚Üí| Field      | Description                                                                              |
  1360‚Üí| :--------- | :--------------------------------------------------------------------------------------- |
  1361‚Üí| `decision` | `"block"` prevents the configuration change from being applied. Omit to allow the change |
  1362‚Üí| `reason`   | Explanation shown to the user when `decision` is `"block"`                               |
  1363‚Üí
  1364‚Üí```json  theme={null}
  1365‚Üí{
  1366‚Üí  "decision": "block",
  1367‚Üí  "reason": "Configuration changes to project settings require admin approval"
  1368‚Üí}
  1369‚Üí```
  1370‚Üí
  1371‚Üí`policy_settings` changes cannot be blocked. Hooks still fire for `policy_settings` sources, so you can use them for audit logging, but any blocking decision is ignored. This ensures enterprise-managed settings always take effect.

  1373‚Üí### WorktreeCreate
  1374‚Üí
  1375‚ÜíWhen you run `claude --worktree` or a [subagent uses `isolation: "worktree"`](/en/sub-agents#choose-the-subagent-scope), Claude Code creates an isolated working copy using `git worktree`. If you configure a WorktreeCreate hook, it replaces the default git behavior, letting you use a different version control system like SVN, Perforce, or Mercurial.
  1376‚Üí
  1377‚ÜíThe hook must print the absolute path to the created worktree directory on stdout. Claude Code uses this path as the working directory for the isolated session.
  1378‚Üí
  1379‚ÜíThis example creates an SVN working copy and prints the path for Claude Code to use. Replace the repository URL with your own:
  1380‚Üí
  1381‚Üí```json  theme={null}
  1382‚Üí{
  1383‚Üí  "hooks": {
  1384‚Üí    "WorktreeCreate": [
  1385‚Üí      {
  1386‚Üí        "hooks": [
  1387‚Üí          {
  1388‚Üí            "type": "command",
  1389‚Üí            "command": "bash -c 'NAME=$(jq -r .name); DIR=\"$HOME/.claude/worktrees/$NAME\"; svn checkout https://svn.example.com/repo/trunk \"$DIR\" >&2 && echo \"$DIR\"'"
  1390‚Üí          }
  1391‚Üí        ]
  1392‚Üí      }
  1393‚Üí    ]
  1394‚Üí  }
  1395‚Üí}
  1396‚Üí```
  1397‚Üí
  1398‚ÜíThe hook reads the worktree `name` from the JSON input on stdin, checks out a fresh copy into a new directory, and prints the directory path. The `echo` on the last line is what Claude Code reads as the worktree path. Redirect any other output to stderr so it doesn't interfere with the path.
  1399‚Üí
  1400‚Üí#### WorktreeCreate input
  1401‚Üí
  1402‚ÜíIn addition to the [common input fields](#common-input-fields), WorktreeCreate hooks receive the `name` field. This is a slug identifier for the new worktree, either specified by the user or auto-generated (for example, `bold-oak-a3f2`).
  1403‚Üí
  1404‚Üí```json  theme={null}
  1405‚Üí{
  1406‚Üí  "session_id": "abc123",
  1407‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1408‚Üí  "cwd": "/Users/...",
  1409‚Üí  "hook_event_name": "WorktreeCreate",
  1410‚Üí  "name": "feature-auth"
  1411‚Üí}
  1412‚Üí```
  1413‚Üí
  1414‚Üí#### WorktreeCreate output
  1415‚Üí
  1416‚ÜíThe hook must print the absolute path to the created worktree directory on stdout. If the hook fails or produces no output, worktree creation fails with an error.
  1417‚Üí
  1418‚ÜíWorktreeCreate hooks do not use the standard allow/block decision model. Instead, the hook's success or failure determines the outcome. Only `type: "command"` hooks are supported.
  1419‚Üí
  1420‚Üí### WorktreeRemove
  1421‚Üí
  1422‚ÜíThe cleanup counterpart to [WorktreeCreate](#worktreecreate). This hook fires when a worktree is being removed, either when you exit a `--worktree` session and choose to remove it, or when a subagent with `isolation: "worktree"` finishes. For git-based worktrees, Claude handles cleanup automatically with `git worktree remove`. If you configured a WorktreeCreate hook for a non-git version control system, pair it with a WorktreeRemove hook to handle cleanup. Without one, the worktree directory is left on disk.
  1423‚Üí
  1424‚ÜíClaude Code passes the path that WorktreeCreate printed on stdout as `worktree_path` in the hook input. This example reads that path and removes the directory:
  1425‚Üí
  1426‚Üí```json  theme={null}
  1427‚Üí{
  1428‚Üí  "hooks": {
  1429‚Üí    "WorktreeRemove": [
  1430‚Üí      {
  1431‚Üí        "hooks": [
  1432‚Üí          {
  1433‚Üí            "type": "command",
  1434‚Üí            "command": "bash -c 'jq -r .worktree_path | xargs rm -rf'"
  1435‚Üí          }
  1436‚Üí        ]
  1437‚Üí      }
  1438‚Üí    ]
  1439‚Üí  }
  1440‚Üí}
  1441‚Üí```
  1442‚Üí
  1443‚Üí#### WorktreeRemove input
  1444‚Üí
  1445‚ÜíIn addition to the [common input fields](#common-input-fields), WorktreeRemove hooks receive the `worktree_path` field, which is the absolute path to the worktree being removed.
  1446‚Üí
  1447‚Üí```json  theme={null}
  1448‚Üí{
  1449‚Üí  "session_id": "abc123",
  1450‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1451‚Üí  "cwd": "/Users/...",
  1452‚Üí  "hook_event_name": "WorktreeRemove",
  1453‚Üí  "worktree_path": "/Users/.../my-project/.claude/worktrees/feature-auth"
  1454‚Üí}
  1455‚Üí```
  1456‚Üí
  1457‚ÜíWorktreeRemove hooks have no decision control. They cannot block worktree removal but can perform cleanup tasks like removing version control state or archiving changes. Hook failures are logged in debug mode only. Only `type: "command"` hooks are supported.

  1487‚Üí
  1488‚ÜíRuns when a Claude Code session ends. Useful for cleanup tasks, logging session
  1489‚Üístatistics, or saving session state. Supports matchers to filter by exit reason.
  1490‚Üí
  1491‚ÜíThe `reason` field in the hook input indicates why the session ended:
  1492‚Üí
  1493‚Üí| Reason                        | Description                                |
  1494‚Üí| :---------------------------- | :----------------------------------------- |
  1495‚Üí| `clear`                       | Session cleared with `/clear` command      |
  1496‚Üí| `logout`                      | User logged out                            |
  1497‚Üí| `prompt_input_exit`           | User exited while prompt input was visible |
  1498‚Üí| `bypass_permissions_disabled` | Bypass permissions mode was disabled       |
  1499‚Üí| `other`                       | Other exit reasons                         |
  1500‚Üí
  1501‚Üí#### SessionEnd input
  1502‚Üí
  1503‚ÜíIn addition to the [common input fields](#common-input-fields), SessionEnd hooks receive a `reason` field indicating why the session ended. See the [reason table](#sessionend) above for all values.
  1504‚Üí
  1505‚Üí```json  theme={null}
  1506‚Üí{
  1507‚Üí  "session_id": "abc123",
  1508‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1509‚Üí  "cwd": "/Users/...",
  1510‚Üí  "permission_mode": "default",
  1511‚Üí  "hook_event_name": "SessionEnd",
  1512‚Üí  "reason": "other"
  1513‚Üí}
  1514‚Üí```
  1515‚Üí
  1516‚ÜíSessionEnd hooks have no decision control. They cannot block session termination but can perform cleanup tasks.
  1517‚Üí
  1518‚Üí## Prompt-based hooks
  1519‚Üí
  1520‚ÜíIn addition to command and HTTP hooks, Claude Code supports prompt-based hooks (`type: "prompt"`) that use an LLM to evaluate whether to allow or block an action, and agent hooks (`type: "agent"`) that spawn an agentic verifier with tool access. Not all events support every hook type.
  1521‚Üí
  1522‚ÜíEvents that support all four hook types (`command`, `http`, `prompt`, and `agent`):
  1523‚Üí
  1524‚Üí* `PermissionRequest`
  1525‚Üí* `PostToolUse`
  1526‚Üí* `PostToolUseFailure`
  1527‚Üí* `PreToolUse`
  1528‚Üí* `Stop`
  1529‚Üí* `SubagentStop`
  1530‚Üí* `TaskCompleted`
  1531‚Üí* `UserPromptSubmit`
  1532‚Üí
  1533‚ÜíEvents that only support `type: "command"` hooks:
  1534‚Üí
  1535‚Üí* `ConfigChange`
  1536‚Üí* `Notification`
  1537‚Üí* `PreCompact`
  1538‚Üí* `SessionEnd`
  1539‚Üí* `SessionStart`
  1540‚Üí* `SubagentStart`
  1541‚Üí* `TeammateIdle`
  1542‚Üí* `WorktreeCreate`
  1543‚Üí* `WorktreeRemove`
  1544‚Üí
  1545‚Üí### How prompt-based hooks work
  1546‚Üí
  1547‚ÜíInstead of executing a Bash command, prompt-based hooks:
  1548‚Üí
  1549‚Üí1. Send the hook input and your prompt to a Claude model, Haiku by default
  1550‚Üí2. The LLM responds with structured JSON containing a decision
  1551‚Üí3. Claude Code processes the decision automatically
  1552‚Üí
  1553‚Üí### Prompt hook configuration
  1554‚Üí
  1555‚ÜíSet `type` to `"prompt"` and provide a `prompt` string instead of a `command`. Use the `$ARGUMENTS` placeholder to inject the hook's JSON input data into your prompt text. Claude Code sends the combined prompt and input to a fast Claude model, which returns a JSON decision.
  1556‚Üí
  1557‚ÜíThis `Stop` hook asks the LLM to evaluate whether all tasks are complete before allowing Claude to finish:
  1558‚Üí
  1559‚Üí```json  theme={null}
  1560‚Üí{
  1561‚Üí  "hooks": {
  1562‚Üí    "Stop": [
  1563‚Üí      {
  1564‚Üí        "hooks": [
  1565‚Üí          {
  1566‚Üí            "type": "prompt",
  1567‚Üí            "prompt": "Evaluate if Claude should stop: $ARGUMENTS. Check if all tasks are complete."
  1568‚Üí          }
  1569‚Üí        ]
  1570‚Üí      }
  1571‚Üí    ]
  1572‚Üí  }
  1573‚Üí}
  1574‚Üí```
  1575‚Üí
  1576‚Üí| Field     | Required | Description                                                                                                                                                         |
  1577‚Üí| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
  1578‚Üí| `type`    | yes      | Must be `"prompt"`                                                                                                                                                  |
  1579‚Üí| `prompt`  | yes      | The prompt text to send to the LLM. Use `$ARGUMENTS` as a placeholder for the hook input JSON. If `$ARGUMENTS` is not present, input JSON is appended to the prompt |
  1580‚Üí| `model`   | no       | Model to use for evaluation. Defaults to a fast model                                                                                                               |
  1581‚Üí| `timeout` | no       | Timeout in seconds. Default: 30                                                                                                                                     |
  1582‚Üí
  1583‚Üí### Response schema
  1584‚Üí
  1585‚ÜíThe LLM must respond with JSON containing:
  1586‚Üí
  1587‚Üí```json  theme={null}
  1588‚Üí{
  1589‚Üí  "ok": true | false,
  1590‚Üí  "reason": "Explanation for the decision"
  1591‚Üí}
  1592‚Üí```
  1593‚Üí
  1594‚Üí| Field    | Description                                                |
  1595‚Üí| :------- | :--------------------------------------------------------- |
  1596‚Üí| `ok`     | `true` allows the action, `false` prevents it              |
  1597‚Üí| `reason` | Required when `ok` is `false`. Explanation shown to Claude |
  1598‚Üí
  1599‚Üí### Example: Multi-criteria Stop hook
  1600‚Üí
  1601‚ÜíThis `Stop` hook uses a detailed prompt to check three conditions before allowing Claude to stop. If `"ok"` is `false`, Claude continues working with the provided reason as its next instruction. `SubagentStop` hooks use the same format to evaluate whether a [subagent](/en/sub-agents) should stop:
  1602‚Üí
  1603‚Üí```json  theme={null}
  1604‚Üí{
  1605‚Üí  "hooks": {
  1606‚Üí    "Stop": [
  1607‚Üí      {
  1608‚Üí        "hooks": [
  1609‚Üí          {
  1610‚Üí            "type": "prompt",
  1611‚Üí            "prompt": "You are evaluating whether Claude should stop working. Context: $ARGUMENTS\n\nAnalyze the conversation and determine if:\n1. All user-requested tasks are complete\n2. Any errors need to be addressed\n3. Follow-up work is needed\n\nRespond with JSON: {\"ok\": true} to allow stopping, or {\"ok\": false, \"reason\": \"your explanation\"} to continue working.",
  1612‚Üí            "timeout": 30
  1613‚Üí          }
  1614‚Üí        ]
  1615‚Üí      }
  1616‚Üí    ]
  1617‚Üí  }
  1618‚Üí}
  1619‚Üí```
  1620‚Üí
  1621‚Üí## Agent-based hooks
  1622‚Üí
  1623‚ÜíAgent-based hooks (`type: "agent"`) are like prompt-based hooks but with multi-turn tool access. Instead of a single LLM call, an agent hook spawns a subagent that can read files, search code, and inspect the codebase to verify conditions. Agent hooks support the same events as prompt-based hooks.
  1624‚Üí
  1625‚Üí### How agent hooks work
  1626‚Üí
  1627‚ÜíWhen an agent hook fires:
  1628‚Üí
  1629‚Üí1. Claude Code spawns a subagent with your prompt and the hook's JSON input
  1630‚Üí2. The subagent can use tools like Read, Grep, and Glob to investigate
  1631‚Üí3. After up to 50 turns, the subagent returns a structured `{ "ok": true/false }` decision
  1632‚Üí4. Claude Code processes the decision the same way as a prompt hook
  1633‚Üí
  1634‚ÜíAgent hooks are useful when verification requires inspecting actual files or test output, not just evaluating the hook input data alone.
  1635‚Üí
  1636‚Üí### Agent hook configuration
  1637‚Üí
  1638‚ÜíSet `type` to `"agent"` and provide a `prompt` string. The configuration fields are the same as [prompt hooks](#prompt-hook-configuration), with a longer default timeout:
  1639‚Üí
  1640‚Üí| Field     | Required | Description                                                                                 |
  1641‚Üí| :-------- | :------- | :------------------------------------------------------------------------------------------ |
  1642‚Üí| `type`    | yes      | Must be `"agent"`                                                                           |
  1643‚Üí| `prompt`  | yes      | Prompt describing what to verify. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
  1644‚Üí| `model`   | no       | Model to use. Defaults to a fast model                                                      |
  1645‚Üí| `timeout` | no       | Timeout in seconds. Default: 60                                                             |
  1646‚Üí
  1647‚ÜíThe response schema is the same as prompt hooks: `{ "ok": true }` to allow or `{ "ok": false, "reason": "..." }` to block.
  1648‚Üí
  1649‚ÜíThis `Stop` hook verifies that all unit tests pass before allowing Claude to finish:
  1650‚Üí
  1651‚Üí```json  theme={null}
  1652‚Üí{
  1653‚Üí  "hooks": {
  1654‚Üí    "Stop": [
  1655‚Üí      {
  1656‚Üí        "hooks": [
  1657‚Üí          {
  1658‚Üí            "type": "agent",
  1659‚Üí            "prompt": "Verify that all unit tests pass. Run the test suite and check the results. $ARGUMENTS",
  1660‚Üí            "timeout": 120
  1661‚Üí          }
  1662‚Üí        ]
  1663‚Üí      }
  1664‚Üí    ]
  1665‚Üí  }
  1666‚Üí}
  1667‚Üí```
  1668‚Üí
  1669‚Üí## Run hooks in the background
  1670‚Üí
  1671‚ÜíBy default, hooks block Claude's execution until they complete. For long-running tasks like deployments, test suites, or external API calls, set `"async": true` to run the hook in the background while Claude continues working. Async hooks cannot block or control Claude's behavior: response fields like `decision`, `permissionDecision`, and `continue` have no effect, because the action they would have controlled has already completed.
  1672‚Üí
  1673‚Üí### Configure an async hook
  1674‚Üí
  1675‚ÜíAdd `"async": true` to a command hook's configuration to run it in the background without blocking Claude. This field is only available on `type: "command"` hooks.
  1676‚Üí
  1677‚ÜíThis hook runs a test script after every `Write` tool call. Claude continues working immediately while `run-tests.sh` executes for up to 120 seconds. When the script finishes, its output is delivered on the next conversation turn:
  1678‚Üí
  1679‚Üí```json  theme={null}
  1680‚Üí{
  1681‚Üí  "hooks": {
  1682‚Üí    "PostToolUse": [
  1683‚Üí      {
  1684‚Üí        "matcher": "Write",
  1685‚Üí        "hooks": [
  1686‚Üí          {
  1687‚Üí            "type": "command",
  1688‚Üí            "command": "/path/to/run-tests.sh",
  1689‚Üí            "async": true,
  1690‚Üí            "timeout": 120
  1691‚Üí          }
  1692‚Üí        ]
  1693‚Üí      }
  1694‚Üí    ]
  1695‚Üí  }
  1696‚Üí}
  1697‚Üí```
  1698‚Üí
  1699‚ÜíThe `timeout` field sets the maximum time in seconds for the background process. If not specified, async hooks use the same 10-minute default as sync hooks.
  1700‚Üí
  1701‚Üí### How async hooks execute
  1702‚Üí
  1703‚ÜíWhen an async hook fires, Claude Code starts the hook process and immediately continues without waiting for it to finish. The hook receives the same JSON input via stdin as a synchronous hook.
  1704‚Üí
  1705‚ÜíAfter the background process exits, if the hook produced a JSON response with a `systemMessage` or `additionalContext` field, that content is delivered to Claude as context on the next conversation turn.
  1706‚Üí
  1707‚Üí### Example: run tests after file changes
  1708‚Üí
  1709‚ÜíThis hook starts a test suite in the background whenever Claude writes a file, then reports the results back to Claude when the tests finish. Save this script to `.claude/hooks/run-tests-async.sh` in your project and make it executable with `chmod +x`:
  1710‚Üí
  1711‚Üí```bash  theme={null}
  1712‚Üí#!/bin/bash
  1713‚Üí# run-tests-async.sh
  1714‚Üí
  1715‚Üí# Read hook input from stdin
  1716‚ÜíINPUT=$(cat)
  1717‚ÜíFILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
  1718‚Üí
  1719‚Üí# Only run tests for source files
  1720‚Üíif [[ "$FILE_PATH" != *.ts && "$FILE_PATH" != *.js ]]; then
  1721‚Üí  exit 0
  1722‚Üífi
  1723‚Üí
  1724‚Üí# Run tests and report results via systemMessage
  1725‚ÜíRESULT=$(npm test 2>&1)
  1726‚ÜíEXIT_CODE=$?
  1727‚Üí
  1728‚Üíif [ $EXIT_CODE -eq 0 ]; then
  1729‚Üí  echo "{\"systemMessage\": \"Tests passed after editing $FILE_PATH\"}"
  1730‚Üíelse
  1731‚Üí  echo "{\"systemMessage\": \"Tests failed after editing $FILE_PATH: $RESULT\"}"
  1732‚Üífi
  1733‚Üí```
  1734‚Üí
  1735‚ÜíThen add this configuration to `.claude/settings.json` in your project root. The `async: true` flag lets Claude keep working while tests run:
  1736‚Üí
  1737‚Üí```json  theme={null}
  1738‚Üí{
  1739‚Üí  "hooks": {
  1740‚Üí    "PostToolUse": [
  1741‚Üí      {
  1742‚Üí        "matcher": "Write|Edit",
  1743‚Üí        "hooks": [
  1744‚Üí          {
  1745‚Üí            "type": "command",
  1746‚Üí            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/run-tests-async.sh",
  1747‚Üí            "async": true,
  1748‚Üí            "timeout": 300
  1749‚Üí          }
  1750‚Üí        ]
  1751‚Üí      }
  1752‚Üí    ]
  1753‚Üí  }
  1754‚Üí}
  1755‚Üí```
  1756‚Üí
  1757‚Üí### Limitations
  1758‚Üí
  1759‚ÜíAsync hooks have several constraints compared to synchronous hooks:
  1760‚Üí
  1761‚Üí* Only `type: "command"` hooks support `async`. Prompt-based hooks cannot run asynchronously.
  1762‚Üí* Async hooks cannot block tool calls or return decisions. By the time the hook completes, the triggering action has already proceeded.
  1763‚Üí* Hook output is delivered on the next conversation turn. If the session is idle, the response waits until the next user interaction.
  1764‚Üí* Each execution creates a separate background process. There is no deduplication across multiple firings of the same async hook.
  1765‚Üí
  1766‚Üí## Security considerations
  1767‚Üí
  1768‚Üí### Disclaimer
  1769‚Üí
  1770‚ÜíCommand hooks run with your system user's full permissions.
  1771‚Üí
  1772‚Üí<Warning>
  1773‚Üí  Command hooks execute shell commands with your full user permissions. They can modify, delete, or access any files your user account can access. Review and test all hook commands before adding them to your configuration.
  1774‚Üí</Warning>
  1775‚Üí
  1776‚Üí### Security best practices
  1777‚Üí
  1778‚ÜíKeep these practices in mind when writing hooks:
  1779‚Üí
  1780‚Üí* **Validate and sanitize inputs**: never trust input data blindly
  1781‚Üí* **Always quote shell variables**: use `"$VAR"` not `$VAR`
  1782‚Üí* **Block path traversal**: check for `..` in file paths
  1783‚Üí* **Use absolute paths**: specify full paths for scripts, using `"$CLAUDE_PROJECT_DIR"` for the project root
  1784‚Üí* **Skip sensitive files**: avoid `.env`, `.git/`, keys, etc.
  1785‚Üí
