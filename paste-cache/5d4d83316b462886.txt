落地不需要复杂的状态追踪。最简的做法：
verify.py 每次跑完把结果写一个文件：
json// ~/.claude/merit/verify_result.json
{
  "file": "merit_gate.py",
  "time": "2026-04-02T20:30:00",
  "results": {
    "syntax": "pass",
    "logic": "pass", 
    "chain": "fail",
    "docs": "fail"
  },
  "pass_count": 2,
  "total": 4
}
mission complete 时石卫读这个文件：

最终结果 4/4 ✅ → 看历史有没有 ❌ 再变 ✅ 的（说明改了几轮）
第一次就 4/4 → +3
改了几轮才 4/4 → +2
最终还有 ❌ → 不让 complete，打回

这样任务质量就被 verify.py 的客观输出量化了。不靠 AI 自己说"做好了"，靠 verify.py 说"4/4 通过了"。AI 骗不了 verify.py。

比如评分第一件事还没评分完成的时候，他们可以继续进行第二件事第二个石卫进行评分，第三，第四13:44识别出异步钩子已支持并行评分能力。识别出异步钩子已支持并行评分能力。不需要多个石卫。只需要把评分改成异步就行。
现在的问题是评分是同步的——石卫调 MiniMax 评分，等 1-2 秒拿到结果，AI 才能继续。
Claude Code 的 hook 本身就支持 async: true。Stop hook 设成异步后：
AI 做完第一件事 → Stop 触发 → 石卫后台调 MiniMax 评分（不等结果）
    ↓
AI 立刻开始第二件事 → Stop 触发 → 另一个后台评分
    ↓
AI 继续第三件事
    ↓
第一件的评分结果回来了 → 写入 credit.json
第二件的评分结果回来了 → 写入 credit.json
AI 不用等，石卫在后台排队处理。结果回来一个写一个。效果跟你说的"多个石卫"一样，但实现上只是一个配置改动——settings.json 里 Stop hook 加 "async": true。
但有一个注意点： 石卫的 PreToolUse（拦截）绝对不能异步——拦截必须是同步的，AI 必须等石卫说"放行"才能执行操作。只有评分（Stop）适合异步，因为评分结果不影响 AI 当前操作。
json{
  "hooks": {
    "PreToolUse": { "async": false },  // 拦截必须同步等结果
    "PostToolUse": { "async": false }, // 审计也同步（verify.py 结果 AI 要看）
    "Stop": { "async": true }          // 评分异步，不阻塞 AI
  }
}
这样 PreToolUse 拦截是实时的（毫秒级，纯 if/else 不调 AI），Stop 评分是后台的（1-2 秒但不阻塞），AI 干活完全不受影响。
