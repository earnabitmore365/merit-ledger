  ---                                                                           
  Slash Command 是什么
                                                                                
  就是一个 .md 文件，放在 ~/.claude/commands/ 里。
  你在任何 Claude Code 对话里输入 /文件名，那个文件的内容就变成一段提示词发给
  AI。

  就这么简单。不需要编程，就是写文字。

  ---
  针对你的问题，我的设计思路

  两个命令配套，解决"你不需要动脑摘要"的问题：

  /plan-output（给 Opus 用）

  Opus 讨论完以后，你输入这个命令，它会按固定格式强制输出完整交接文档，包含：
  - 已完成的内容
  - 核心范式转变（"为什么"，不只是"做什么"）
  - 技术决策和原因
  - 约束条件
  - 下一步任务清单
  - 验证标准
  - 关键文件路径

  你复制这份文档，原封不动贴给 Sonnet。

  /plan-execute（给 Sonnet 用）

  你贴完文档后输入这个命令，Sonnet 会：
  1. 先输出"我的理解是……"（逼它把理解显式化）
  2. 列出它识别到的关键约束
  3. 列出它识别到的范式/思路
  4. 问你"有没有遗漏或偏差？"
  5. 确认后再开始执行

  ---
  好处

  - 你不用摘要，Opus 自己生成完整文档
  - Sonnet 不能跳过理解直接执行，先验证再动手
  - 格式固定，每次交接都一致

  ---
  真正的解法：项目内的活文档

  在项目里维护一个
  CHECKPOINT.md，随时反映当前状态。不是固定格式，而是固定几个关键维度：

  - 目标：我们在解决什么问题
  - 已完成：做了什么，结论是什么
  - 关键决策：为什么这么做（范式、方向）
  - 约束：不能做什么，为什么
  - 当前状态：进行到哪一步了
  - 下一步：谁要做什么

  每次交接前，当前的 AI 更新这个文件。接手的 AI 第一件事就是读这个文件。

  好处：
  - 没有格式限制，内容随复杂度自然增长
  - 有 git 历史，可以回溯
  - 任何方向的交接都适用
  - 你也能看到当前状态

  ---
  Slash Command 的作用变了

  不是"固定格式模板"，而是一个触发器：

  - /checkpoint — 提示当前 AI："按那几个维度，把当前状态写入 CHECKPOINT.md"
  - 接手的 AI 收到后，先读文件，再开始工作

  这样你要做的就只是：交接前说 /checkpoint，就完了。
