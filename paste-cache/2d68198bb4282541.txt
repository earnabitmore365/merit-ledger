⏺ 找到根因了。

  handle_stop（行 1567-1569）调用 check_channel(cwd)，但 cwd 的传入方式有问题：

  def handle_stop(data):
      cwd = data.get("cwd", "")
      check_channel(cwd)

  check_channel(cwd) 收到的是字符串 cwd，但行 1124：
  me = determine_agent(cwd) if isinstance(cwd, str) else determine_agent({"cwd": cwd})

  这会把 cwd 字符串传给 determine_agent。让我看 determine_agent 期望什么：

  Searched for 1 pattern, read 1 file (ctrl+o to expand)

⏺ determine_agent 能处理字符串，没问题。那问题不在这里。

  回到最初的问题——mtime <= last_mtime 返回了 True，通道已标记为已读。但什么时候标记的？
  SessionStart 时 inject_channel_messages 读了通道后就标记了 last_mtime。所以 Stop hook
  的 check_channel 跑的时候，mtime 已经等于 last_mtime，直接 return 了。

  但执事说"stop hook没有触发"——不是"通道没显示"。让我想想 Stop hook 是否真的被调用了：

  Searched for 1 pattern (ctrl+o to expand)

⏺ Stop hook 一直在触发（最近到 09:32）。所以不是"Stop hook 没触发"，而是 Stop hook 里的
  check_channel 因为 mtime 已被 SessionStart 标记为已读，所以没显示通道消息。

  但执事的问题可能是：为什么这个 session 启动时 SessionStart 的通道提醒没有显示给我看？

  回到 inject_channel_messages——它在 SessionStart
  时跑了，但可能太极写通道的时间戳早于上一次 SessionStart
  的检查时间。也就是说，太极写了通道消息后，两仪的 session 没有重新启动过（一直在同一个
  session 里），所以 SessionStart 没再跑。Stop hook 里的 check_channel 也因为
  SessionStart 已标记已读而跳过。

  汇报：

  Stop hook 有触发（shiwei_log 确认到 09:32）。问题是通道消息没显示，原因：

  1. 太极 09:30 写了通道消息
  2. 但两仪这个 session 从更早就开始了，SessionStart 只在 session 启动时跑一次
  3. Stop hook 的 check_channel 在每次 Stop 时跑，但如果 SessionStart 已经把 last_mtime
  标记为已读（从更早的 session 启动），新消息的 mtime 就会 <= last_mtime
  4. 实际检查：last_mtime = 1775345452.47 == channel mtime = 1775345452.47，完全相等 →
  返回已读

  根因：SessionStart 标记了已读，但太极后来在同一个 session 期间写了新消息。Stop hook 的
   check_channel 用 mtime 判断，mtime 一致就跳过。

  疑问：如果太极 09:30 写了新消息，channel 文件的 mtime 应该更新才对。除非 mtime
  精度问题或者写入没改变 mtime。
